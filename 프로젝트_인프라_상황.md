## 왜 지금 AWS를 아직 안 만들었는가

### 1. OCR 흐름이 먼저다

- 지금 가장 중요한 것은 `OCR 품질`과 `문서 분류 흐름` 검증이다.
- AWS 리소스를 먼저 만들면 빨라 보이지만, 실제로는 OCR 공급자나 비동기 처리 구조가 바뀌면 다시 손대야 한다.

### 2. 현재는 기능 검증 단계다

- 지금 단계는 운영 인프라보다 `실제로 문서를 넣었을 때 결과가 어떻게 나오는지`를 확인해야 한다.
- 문서 8종과 여권에서 어떤 필드가 안정적으로 나오는지 먼저 봐야 한다.

### 3. 아키텍처 결정은 어느 정도 끝났지만, 구현 시점은 아직 아니다

- AWS 방향은 거의 정해졌다.
- 다만 지금 바로 리소스를 생성하기보다, 내일 기준으로는 `구조 확정 + 공급자 선택 + worker 분리 순서`가 더 합리적이다.

## AWS 목표 아키텍처

## 추천 구조

```text
사용자
  -> ALB
  -> admin-web (ECS Fargate)
  -> admin-api (ECS Fargate)

파일 업로드
  -> S3 raw-documents
  -> S3 Event Notification
  -> SQS ocr-jobs
  -> ocr-worker (ECS Fargate)
  -> OCR / 분류 / MRZ / 검증
  -> RDS PostgreSQL 저장
  -> 결과 상태 결정 (READY / REVIEW_REQUIRED)

파생 산출물
  -> S3 derived-artifacts
```

## 왜 이 구조를 추천하는가

### ECS Fargate

- Spring Boot 백엔드를 컨테이너로 올리기 가장 자연스럽다.
- EKS보다 운영 복잡도가 낮다.
- 현재 규모에서는 충분히 감당 가능하다.

### S3

- 원본 파일과 파생 파일을 분리 저장하기 좋다.
- DB에 문서 바이너리를 직접 넣는 것보다 훨씬 단순하다.
- 미리보기, crop 이미지, OCR 스냅샷 저장에도 유리하다.

### SQS + OCR Worker

- OCR은 업로드 API 안에서 바로 처리하면 느려지고 실패 처리도 어렵다.
- 비동기 큐로 분리하면 재시도와 장애 대응이 쉬워진다.
- 운영 단계에서는 반드시 `admin-api`와 `ocr-worker`를 분리하는 게 맞다.

### RDS PostgreSQL

- 문서 메타데이터, OCR 결과, 검수 상태 관리에 적합하다.
- 지금 DDD 구조와도 잘 맞는다.

### Secrets Manager

- OCR 키와 DB 비밀번호를 앱 설정 파일에 넣지 않기 위해 필요하다.
- 현재 로컬의 `application-local.yml` 역할을 AWS에서는 Secrets Manager가 대신하게 된다.

### ALB + Cognito 또는 사내 OIDC

- 내부 운영툴은 앱 코드에서 로그인 기능을 크게 키우기보다, 앞단 인증으로 처리하는 편이 단순하다.
- 나중에 권한 체계가 커져도 확장하기 쉽다.

## 아직 안 된 것

### 1. AWS 리소스 생성

- 아직 VPC, ECS, S3, SQS, RDS를 만들지 않았다.

이유:

- 지금은 코드/흐름 검증이 우선이라 인프라 선구축의 이점이 작다.

### 2. 비동기 worker 분리

- 현재는 백엔드 내부 서비스 흐름에서 OCR을 처리한다.

이유:

- 1차는 빠른 검증이 목적이었다.
- 처음부터 worker, queue, 상태머신까지 다 넣으면 구현 속도가 느려진다.

### 3. 운영 DB 연결

- 아직 운영용 PostgreSQL 연결 전이다.

이유:

- 현재는 기능 구조를 먼저 잡는 단계라서, 저장소 세부 구현은 뒤로 미뤘다.

### 4. OCR 공급자 최종 확정

- OpenAI 어댑터와 Google Vision 어댑터는 모두 코드로 붙여놓았다.
- 하지만 어떤 공급자를 메인으로 쓸지는 아직 최종 결정 전이다.

이유:

- 실제 샘플 문서 기준으로 `정확도`, `비용`, `분류 안정성`을 비교해야 한다.

이유:

  - SQS + DLQ
