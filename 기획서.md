# OCR 비자 신청 시스템 백엔드 기획서

작성일: 2026-05-08

## 1. 목적

이 문서는 현재 프론트 요구사항을 기준으로, 백엔드에서 앞으로 어떤 도메인과 데이터 구조가 필요할지 정리하기 위한 초안이다.  
지금 단계에서는 코드를 바로 추가하기보다, `어떤 데이터가 존재해야 하는지`, `OCR 결과를 어떤 순서로 조합해야 하는지`, `나중에 어떤 테이블이 필요할지`를 먼저 고정하는 것이 목적이다.

## 2. 현재 단계 정의

- 백엔드 코드는 이미 OCR 어댑터와 기본 처리 흐름 초안이 존재한다.
- 하지만 실제 업무 도메인 기준의 신청 케이스 관리 구조는 아직 시작 전이다.
- 따라서 지금은 `도메인 정리 전 기획 단계`로 본다.

## 5. 핵심 도메인 해석

### 학생

- 학생은 사람 자체를 의미한다.
- 한 학생은 여러 신청건을 가질 수 있다.
- 따라서 학생과 신청 케이스는 분리해야 한다.

### 신청 케이스

- 실제 업무 단위는 학생이 아니라 `신청 건`이다.
- 화면도 유학원에서는 신청 건 단위로 보게 된다.
- 따라서 리스트, 상태, 검수, 제출 서류는 케이스 단위로 관리하는 것이 맞다.

### 비자 유형

- 비자 유형이 필요 서류 집합을 결정한다.
- 따라서 `비자 유형 -> 필요 서류 정의` 매핑 구조가 필요하다.

### 제출 서류

- 필요 서류와 제출된 서류는 구분해야 한다.
- 필요 서류는 템플릿이고, 제출 서류는 실제 파일이다.

## 6. ZIP 업로드 및 OCR 조립 규칙

이 부분은 나중에 OCR 파이프라인과 케이스 생성 규칙의 핵심이다.

### 학생 구간 분리 규칙

- ZIP 내부 스캔본을 순서대로 본다.
- `통합신청서`가 나오면 새 학생 세트의 시작으로 본다.
- 다음 `통합신청서`가 나오기 전까지는 같은 학생의 문서 묶음이다.

즉:

- 통합신청서 1개 = 학생 구간 시작 신호
- 다음 통합신청서 전까지 = 같은 학생 문서 묶음

### 학생 정보 추출 우선순위

1. 여권
   - 이름
   - 국적
   - 생년월일
2. 통합신청서
   - 여권에서 불완전한 값 보완
   - 외국인등록번호
   - 전화번호
3. 부동산계약서
   - 주소

### 정리된 처리 순서

1. ZIP 업로드
2. 파일 목록 정렬
3. 통합신청서 기준 학생 구간 분리
4. 각 구간 안에서 문서 유형 분류
5. 여권에서 기본 인적사항 우선 추출
6. 통합신청서에서 누락값 및 추가값 보완
7. 부동산계약서에서 주소 추출
8. 비자 유형별 필요 서류 충족 여부 계산
9. 케이스 상태 결정 또는 검수 대기

- 실제 업로드된 문서 파일

### 7. upload_batch

- ZIP 업로드 단위
- 누가 업로드했는지

- 미제출

### 9. ocr_extraction_result

- 문서별 OCR 원문
- 추출 필드
- confidence
- provider

### 10. review_task

- 사람이 검수해야 하는 작업
- 어떤 문서/필드가 불확실한지 저장

## 8. 왜 이런 분리가 필요한가

### student 와 application_case 분리 이유

- 한 학생이 여러 번 신청할 수 있기 때문이다.
- 화면도 학생 자체보다 신청 건 중심으로 움직인다.

### requirement 와 submitted_document 분리 이유

- 필요 서류 목록은 비자 유형 기준의 정의 데이터다.
- 제출 서류는 실제 업로드 결과다.
- 둘을 섞으면 누락 계산과 검수 로직이 복잡해진다.

### upload_batch 분리 이유

- ZIP 파일 1개 안에 여러 학생 자료가 들어간다.
- 따라서 파일 업로드 단위와 신청 케이스 단위는 다르다.

## 14. 결론

지금은 백엔드 코드를 더 늘릴 시점이 아니라, 어떤 데이터를 중심으로 서비스가 움직일지를 먼저 고정해야 하는 단계다.  
핵심 축은 `학생`이 아니라 `신청 케이스`, 핵심 자동화 규칙은 `통합신청서 기준 분리 + 여권 우선 추출 + 통합신청서 보완 + 부동산계약서 주소 추출`이다.  
이 구조가 정리되면 그다음 도메인 설계와 AWS 인프라 설계를 훨씬 안정적으로 진행할 수 있다.
